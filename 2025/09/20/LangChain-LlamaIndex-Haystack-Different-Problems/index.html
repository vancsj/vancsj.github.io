LangChain, LlamaIndex, and Haystack Are Not Competing

发表于 2025-09-20 阅读次数：

LLMs know everything in their training data and nothing about yours. They’ve never seen your company’s docs, your database, or last week’s incident reports. Getting that private data into the model at query time — the right chunks, in the right order, under the token limit — is what context engineering actually is.

大语言模型知道训练数据里的一切，却对你的数据一无所知。它没见过你公司的文档、你的数据库、上周的故障报告。所谓上下文工程，就是在查询时把正确的私有数据注入模型——合适的片段、合适的顺序、不超token限制。

Three frameworks dominate this space: LangChain, LlamaIndex, Haystack. They’re not competing to solve the same problem. Each one made a different architectural bet.

这个领域有三个主流框架：LangChain、LlamaIndex、Haystack。它们不是在争同一个赛道，而是各自押注了不同的架构方向。

The pipeline for feeding an LLM private data: load documents → chunk → vectorize → store → at query time, retrieve the most relevant chunks → pass to the LLM. LlamaIndex was built specifically for this pipeline and nothing else.

给LLM注入私有数据的流程大致是：加载文档 → 切片 → 向量化 → 存储 → 查询时检索最相关片段 → 传给LLM。LlamaIndex就是为这条流水线而生的，专注于此，不做别的。

LlamaParse handles the messy cases — PDFs with tables, images, irregular layouts. If your bottleneck is “get my private documents into a queryable index,” LlamaIndex is the fastest path there.

LlamaParse处理那些麻烦的格式——带表格、图片、乱排版的PDF。如果你的问题就是”怎么把私有文档做成可查询的索引”，LlamaIndex是最快的路径。

The limit: once you need an agent that decides between querying a database, hitting an API, or pulling a document, LlamaIndex runs out of road.

它的局限：一旦你需要一个能在数据库查询、API调用、文档检索之间自己决策的代理，LlamaIndex就不够用了。

LangChain’s core abstraction is the chain — a sequence of LLM calls, tool uses, and memory reads, composed together. The framework is built for apps where the LLM has to decide what to do next: pick a tool, route a request, maintain state across turns.

LangChain的核心抽象是”链”——把LLM调用、工具调用、记忆读取串联起来。这个框架是为需要LLM自己决定下一步的应用设计的：选工具、路由请求、跨轮次维护状态。

The tradeoff is abstraction overhead. LangChain has a lot of layers, and when something breaks, the stack trace is deep. LangSmith — their observability tool — exists because debugging LangChain apps without it is genuinely painful.

代价是抽象层带来的复杂度。LangChain层很多，出了问题调用栈很深。他们专门做了LangSmith这个可观测性工具，因为没有它调试LangChain应用真的很难受。

Use LangChain when the complexity is in the orchestration: multi-step reasoning, agent loops, or workflows that chain several LLM calls with tool use in between.

如果复杂度在编排层——多步推理、代理循环、夹杂着工具调用的多轮LLM交互——用LangChain。

Haystack’s bet was on production reliability and measurement. It’s built around an explicit pipeline architecture: a Retriever finds candidate docs, a Reader extracts the answer, and each stage is independently testable. You can benchmark recall, precision, and latency at each step before shipping.

Haystack押注的是生产可靠性和可度量性。它的核心是显式的流水线架构：Retriever找候选文档，Reader提取答案，每个阶段都能独立测试。上线之前可以对每一步的召回率、准确率、延迟分别跑基准测试。

The hybrid search support — BM25 keyword matching combined with vector similarity — is the most mature of the three. For enterprise search where you need to prove accuracy requirements before handing the system to users, Haystack has the tooling.

它的混合搜索——BM25关键词匹配加上向量相似度——是三者里最成熟的。如果你在做企业搜索，需要在交付给用户之前证明系统达到了准确率要求，Haystack有这套工具。

The community is smaller than LangChain’s but the documentation is better. That gap suggests something about the intended user: engineers who want to understand the system, not just get it running.

社区比LangChain小，但文档更好。这个差距说明了它的目标用户：想搞清楚系统原理的工程师，而不是把它跑起来就行。

These frameworks are not mutually exclusive. A common production pattern: LlamaIndex handles document ingestion and indexing, then the retrieved context passes into a LangChain agent for multi-step reasoning. You get LlamaIndex’s RAG depth and LangChain’s orchestration in the same system.

这三个框架可以组合用。一种常见的生产模式：用LlamaIndex做文档摄取和索引，检索到的上下文再传给LangChain代理做多步推理。RAG深度和编排灵活性两者兼得。

The real split is about what problem dominates your architecture. Pure RAG over documents: LlamaIndex. Complex agent workflows: LangChain. Enterprise search with measurement requirements: Haystack.

选择的依据是你的架构里哪个问题最核心。纯文档RAG用LlamaIndex，复杂代理工作流用LangChain，有度量要求的企业搜索用Haystack。
