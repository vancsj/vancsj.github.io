AI Agents and Software Engineering: What's Actually Changing

发表于 2025-02-01 阅读次数：

The first AI coding tools were smart autocomplete — you typed, they suggested. That was the whole loop.

第一批AI编程工具就是智能补全：你打字，它给建议，仅此而已。

Agents are different. An agent perceives its environment, decides what to do next, acts, and adjusts based on what it sees. It has goals, not just inputs.

代理不一样。它有目标，不只是响应输入。它会感知环境、决策、行动，再根据结果调整方向。

The practical gap shows up when things go wrong: a chatbot fails and stops; an agent fails and tries something else.

区别在出错的时候最明显——聊天机器人卡死就算了，代理换个方法继续。

Four things make an agent work.

代理由四个部分构成。

Perception — reads the environment. Code files, error messages, API responses.

感知 — 读取环境：代码文件、报错信息、API响应。

Reasoning — an LLM figures out what to do next. This is the hard part, and why LLMs are central to agent design.

推理 — 大模型决定下一步怎么做。这是最难的部分，也是为什么LLM是代理架构的核心。

Action — actually does something. Writes a file. Calls an API. Runs a test.

行动 — 真的去做：写文件、调接口、跑测试。

Memory — short-term (current context) and long-term (what it learned across sessions). Most current tools only have perception and action. Memory is the unsolved part.

记忆 — 分短期（当前上下文）和长期（跨会话的积累）。目前多数工具有感知和行动，记忆还是个悬案。

Today’s agents work like batch jobs — input goes in, output comes out, everything is forgotten. Fine for simple tasks, but they break down on anything that requires remembering what was tried, what failed, and why.

现在的代理本质上是无状态的：一次请求，一次响应，上下文清零。做个简单任务没问题，但稍微复杂一点——需要记住试过什么、哪里出了问题——就撑不住了。

The real shift will be stateful agents that can pick up a problem across sessions and maintain something like project knowledge.

真正的突破是有状态的代理：能跨会话接着干，慢慢积累对项目的理解，知道之前试了什么、为什么卡住。

This is why agent memory architecture matters more than model capability right now. The models are already good enough for many tasks; the bottleneck is persistence.

所以现在记忆架构比模型本身更重要。模型的能力已经够用了，真正的瓶颈是怎么让它记住东西。

For software engineers, the work is changing — not “write code” anymore, but “design the systems where an agent can write good code reliably.”

对工程师来说，工作重心变了。不再是亲自写代码，而是搭出一个让代理能可靠运作的系统。

The work shifts upstream. You define module boundaries, document architecture conventions, build specialized tools for specific domains. The agent handles mechanical parts; you handle the parts that require understanding why the system is structured the way it is.

工作向上游移动：定义模块边界、记录架构约定、为特定领域搭专用工具。机械性的活交给代理，你来处理那些需要理解”系统为什么这样设计”的部分。

This is specialization, not replacement. Senior engineers don’t write boilerplate — they design systems where others can write safe boilerplate. Agents fit a similar slot.

这是分工，不是替代。高级工程师从来不亲自写样板代码，他们设计让别人能安全写样板代码的系统。代理的角色和这差不多。

Three tools dominate agentic CLI coding right now: Aider, Gemini CLI, Claude Code. Three distinct philosophies.

目前代理式命令行编程有三个主流工具：Aider、Gemini CLI、Claude Code，设计哲学各不相同。

Aider builds a symbol map of your repo and makes edits in diff format. It verifies the original code block exists before applying any change. Reliable, transparent, good for targeted edits.

Aider 先建代码库的符号索引，改动以diff格式输出，每次修改前都确认原始代码块存在。可靠、透明，适合需要精准改动的场景。

Gemini CLI explores the codebase on demand — grep, file reads, piecing together context as it goes. Works well for multi-file tasks where you can’t predict upfront what context you’ll need.

Gemini CLI 按需探索代码库——grep、读文件、边走边拼凑上下文。适合那种改之前根本不知道要看哪些文件的多文件任务。

Claude Code treats editing as one step in a larger workflow — file manipulation combined with hooks for linters and tests, project-specific memory, and pre-planned execution. Better for delegating entire workflows.

Claude Code 把编辑只当成工作流的一步，配合lint/测试钩子、项目级记忆和执行前规划。适合把整个任务流程都交给它的情况。

Pick by task. Precise edit: Aider. Exploratory multi-file work: Gemini CLI. Full workflow automation: Claude Code.

按任务选。精准改动用Aider，探索性多文件任务用Gemini CLI，完整工作流自动化用Claude Code。

Code completion is still mostly bad. Worth saying clearly.

代码补全现在还是差。不客气地说。

The core problem: completion models see the current file. They don’t understand conventions elsewhere in the codebase, constraints set in other modules, or the reason the code is structured the way it is.

核心问题在于：补全模型只看眼前这个文件，不知道代码库其他地方的约定、其他模块的约束，更不知道代码为什么这样设计。

The suggestions that fail most: naming (too generic), patterns (doesn’t match established conventions), context (correct in isolation, wrong given system constraints).

最常出问题的：命名（太通用）、写法（不符合既有约定）、上下文（孤立看没错，但违反系统约束）。

What helps: larger context windows, RAG over the full codebase, fine-tuning on specific projects. None are fully solved at the tool level yet. If you find completion annoying, that’s an accurate read of the current state.

有帮助的方向：更大的上下文窗口、对整个代码库做RAG、在特定项目上微调。但工具层面都还没完全解决。如果你觉得补全很烦，那是对的，工具还没到位。

The long-term pattern matters more than any specific tool.

工具会变，但背后的模式是不变的。

AI-first software engineering means building systems where agents can operate reliably: clear module boundaries, explicit conventions documented somewhere agents can read, deterministic components handling the parts where LLMs make mistakes.

AI优先的软件工程，核心是搭出让代理能可靠工作的系统：模块边界清晰、约定明确记录（放在代理能读到的地方）、用确定性组件兜住LLM容易出错的地方。

The engineers who do well will understand both sides: what LLMs are good at (pattern matching, code generation, summarization) and where they fail (counting, precise constraint tracking, long-range consistency). Design around both.

真正做得好的工程师，既清楚LLM擅长什么（模式匹配、生成、归纳），也清楚它在哪里出问题（计数、精确追踪约束、长程一致性）——然后围绕这两点设计系统。
