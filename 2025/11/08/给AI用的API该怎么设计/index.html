给AI用的API该怎么设计

发表于 2025-11-08 阅读次数：

过去二十年，API设计的核心追求是消除歧义。严格的类型、明确的契约、可预测的响应。但AI带来了一个悖论：AI擅长理解歧义，却也需要更明确的指引。

传统API和AI调用API的本质区别

传统API面向人类开发者。人类会读文档、查Stack Overflow、根据错误信息调试。

AI调用API时没有这些能力。它只有当前的响应和有限的上下文。这意味着：

1. 错误信息要更易读

AI不知道”INVALID_PARAM”意味着什么。它需要具体的提示：哪里错了、怎么修。

2. 响应要告诉AI下一步做什么

这增加了响应大小，但减少了AI调用错误接口或误解数据的概率。

3. 数据结构要自解释

传统API依赖文档解释字段含义。AI不会去读文档。

temp 是摄氏还是华氏？hum 是百分比还是小数？AI猜错了就是你承担后果。

探讨性设计：语义层和协议层

这是探索性的想法，不是生产经验。

传统API设计关注”契约”——接口签名、参数类型、返回格式。AI时代的API可能需要一个”协议层”——描述数据的含义和使用方式。

语义层包含什么：

  • 字段的业务含义（不只是数据类型）
  • 数据的时效性和置信度
  • 字段之间的关联关系
  • 常见的误读提醒

协议层包含什么：

  • 基于当前结果，AI可能想调用的下一个接口
  • 处理当前数据的推荐方式
  • 错误恢复路径

Token成本的权衡

每个token都有成本。添加语义提示会让响应变大。

这是个权衡：多花的token成本 vs AI少犯错的收益。

对于高频、低风险的API，可能不值得。对于低频、高风险的操作（支付、权限变更），多几个token的提示可能避免严重的错误。

MCP

Model Context Protocol是AI领域的”USB-C”——标准化的连接方式。如果你在设计面向AI的系统，值得了解。但这是另一个话题，这里不展开。

核心观点

传统API假设消费者是人类，会自己找答案。AI时代的API要假设消费者是AI，需要在响应中提供足够的上下文和指引。

这不是技术升级，是设计思路的转变。从”让人类能读懂”到”让AI不会读错”。
