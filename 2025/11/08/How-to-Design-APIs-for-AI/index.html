How to Design APIs for AI

发表于 2025-11-08 阅读次数：

For twenty years, API design has been about eliminating ambiguity. Strict types, explicit contracts, predictable responses. But AI introduces a paradox: AI is good at understanding ambiguity, yet it needs clearer guidance than humans do.

The Real Difference: Traditional API vs AI-Calling API

Traditional APIs are designed for human developers. Humans read documentation, search Stack Overflow, debug from error messages.

AI calling an API doesn’t have these abilities. It only has the current response and limited context. This means:

1. Errors need to be readable

AI doesn’t know what “INVALID_PARAM” means. It needs specific guidance: what’s wrong, how to fix it.

2. Responses should tell AI what to do next

This increases response size, but reduces the chance of AI calling wrong endpoints or misinterpreting data.

3. Data structures should be self-explanatory

Traditional APIs rely on documentation to explain field meanings. AI doesn’t have patience for documentation.

Is temp Celsius or Fahrenheit? Is hum a percentage or decimal? If AI guesses wrong, you pay the price.

Exploratory Design: Semantic Layer and Protocol Layer

This is an exploratory idea, not production experience.

Traditional API design focuses on “contracts” — interface signatures, parameter types, return formats. AI-era APIs might need a “protocol layer” — describing what data means and how to use it.

What the semantic layer includes:

  • Business meaning of fields (not just data types)
  • Data freshness and confidence levels
  • Relationships between fields
  • Common misinterpretation warnings

What the protocol layer includes:

  • Next endpoints AI might want to call based on current result
  • Recommended ways to process current data
  • Error recovery paths

Token Cost Trade-off

Every token costs money. Adding semantic hints makes responses larger.

It’s a trade-off: extra token cost vs benefit of fewer AI mistakes.

For high-frequency, low-risk APIs, it might not be worth it. For low-frequency, high-risk operations (payments, permission changes), a few extra tokens of hints might prevent serious errors.

MCP

Model Context Protocol is the “USB-C” for AI — a standardized connection method. If you’re designing AI-facing systems, it’s worth knowing about. But that’s another topic.

The Core Point

Traditional APIs assume human consumers who can figure things out themselves. AI-era APIs should assume AI consumers who need enough context and guidance in the response itself.

This isn’t a technical upgrade. It’s a shift in design thinking. From “make it readable by humans” to “make it un-misinterpretable by AI.”
