MCP vs Skills: 工具箱与菜谱的较量

发表于 2026-04-11 阅读次数：

Last month, a Chinese article asked whether MCP had already been abandoned. I fact-checked the claims, and the real answer turned out to be more interesting: MCP’s limits are real, but the missing piece is probably Skills.

上个月，一篇中文文章在问：MCP是不是已经被抛弃了。我把里面的说法都核查了一遍，最后发现重点不是“MCP死没死”，而是它的短板确实摆在那儿，而补上这块短板的，很可能就是Skills。

So let’s make the comparison simple: what does MCP actually give you, and what do Skills add on top?

所以我们不绕弯子，直接拆开看：MCP到底给你什么，Skills又帮你补什么？

The Claim That Started It All | 引发争议的论点

The Chinese article made several bold claims: 3 MCP servers consuming 14,300 out of 200,000 tokens (72%), Perplexity CTO Denis Yarats abandoning MCP, and MCP being a “blunt instrument” compared to CLI’s “precision.”

那篇中文文章抛出了几个挺猛的说法：3个MCP服务器吃掉了200,000个token里的14,300个（72%）、Perplexity的CTO Denis Yarats放弃了MCP、MCP比CLI更像“粗糙的工具”，而CLI像“精密仪器”。

The fact-check results were mixed. Denis Yarats did indeed launch Ask 2026 without MCP support in March 2026. The 72% context claim is real — multiple sources confirm that loading multiple MCP tools can consume significant tokens. But “tool selection accuracy dropping from 43% to 14%” wasn’t from a specific study — it was the author’s illustrative example.

事实核查下来，结论是有真有假。Denis Yarats确实在2026年3月推出了不支持MCP的 Ask 2026。72%上下文这个说法也站得住脚，多个来源都确认加载多个MCP工具会吃掉不少token。只有“工具选择准确率从43%降到14%”这组数字，不是出自某个正式研究，而是作者举的例子。

Here’s what the fact-check confirmed:

核查后能确认的，大概就是这些：

  • ✅ Perplexity CTO abandoned MCP (Ask 2026, March 2026)

  • ✅ 3 MCP servers = 72% of 200K token context (verified)

  • ✅ MCP donated to Linux Foundation (real)

  • ✅ 10K+ active servers on Glama.ai (20,000+ actually)

  • ⚠️ The specific “43% to 14%” numbers weren’t from a published study

  • ✅ Perplexity CTO放弃了MCP（Ask 2026，2026年3月）

  • ✅ 3个MCP服务器 = 200K token上下文的72%（已验证）

  • ✅ MCP捐给了Linux基金会（真实）

  • ✅ Glama.ai上有10K+活跃服务器（实际有20,000+）

  • ⚠️ “43%到14%”的具体数字并非来自已发表的研究

The debate about MCP being “dead” or “abandoned” misses the point. What’s more interesting is understanding what MCP actually is, and what it’s not.

关于MCP”已死”或”被抛弃”的争论忽略了一个要点。更重要的是理解MCP到底是什么，以及不是什么。

What MCP Actually Gives You | MCP实际上给你什么

MCP (Model Context Protocol) is a tool protocol built on JSON-RPC 2.0. When you connect an MCP server, Claude gets a tool definition and not much else:

MCP（模型上下文协议）本质上是个基于 JSON-RPC 2.0 的工具协议。你连上一个 MCP server 之后，Claude 拿到的基本就是一个工具名、一个描述和一份参数定义，别的就不多了：

That’s it. Just the function signature. “Here’s a screwdriver — it can turn screws.”

就这么多。说白了，就是一个函数签名。”这是一把螺丝刀——能拧螺丝，但不会告诉你什么时候该用它。”

The tool definition includes:

工具定义里通常就这些：

  • Name (label) | 名称（标签）
  • Description (brief summary, ~200 char limit in UI) | 描述（简短摘要，UI中约200字符限制）
  • Input schema (parameters and types) | 输入模式（参数和类型）

What’s NOT included:

但不包括这些：

  • When to use this tool vs alternatives | 什么时候用这个工具而不是其他选择
  • How to compose with other tools | 如何与其他工具组合使用
  • Error handling strategies | 错误处理策略
  • Business logic or team conventions | 业务逻辑或团队约定

The MCP spec itself imposes no hard limit, but practical constraints exist:

MCP 规范本身没有硬性限制，但实际用起来还是有约束：

  • Tool description: ~200 characters (Claude Desktop truncates) | 工具描述：约200字符（Claude Desktop会截断）
  • Input schema total: ~64KB (token budget + transport buffer) | 输入模式总计：约64KB（token预算+传输缓冲）

The design is intentional. MCP is an API spec, not documentation. The “how” belongs in prompts, context, or — as we’ll see — Skills.

这是故意这么设计的。MCP 是 API 规范，不是文档。至于“怎么用”，要放在提示词、上下文里，或者——你等下会看到——交给 Skills。

What Skills Actually Give You | Skills实际上给你什么

Skills are Claude Code’s way of packaging procedural knowledge — not just what tools exist, but how to use them. Here’s a real example from a GitHub skill:

Skills 是 Claude Code 用来装“流程说明”的地方——不只是告诉你有哪些工具，还会把什么时候用、怎么走流程一并写进去。下面是一个 GitHub skill 的真实例子：

This isn’t just a function signature. It includes:

这不仅仅是函数签名。它包括：

  • When to invoke (trigger conditions) | 何时调用（触发条件）
  • How to execute (step-by-step procedure) | 如何执行（具体的操作流程）
  • Why it matters (purpose/context) | 为什么要这么做（背后的逻辑和初衷）

The key difference: Skills are recipes, MCP is a toolbox.

关键区别：Skills是菜谱，MCP是工具箱。

MCP gives you:

MCP提供：

  • External system connections (GitHub, databases, APIs) | 外部系统连接（GitHub、数据库、API）
  • Standardized interface definitions | 标准化的接口定义
  • OAuth 2.1 authentication | OAuth 2.1认证

Skills give you:

Skills提供：

  • Team-specific workflows | 团队特定的工作流
  • Step-by-step procedures | 标准作业程序（SOP）
  • Intelligent tool usage guidance | 智能的工具使用指导

The Real Insight: They Complement Each Other | 真正的洞见：它们互补

Here’s what most discussions miss: MCP and Skills aren’t competitors. They solve different problems.

大多数讨论都漏掉了一点：MCP 和 Skills 不是对手，它们解决的是两类不同的问题。

The practical recommendation from the research:

研究里给出的实际建议是：

“Most developers need: 2-3 MCP servers (GitHub, Filesystem, one domain-specific) + a few custom Skills for their workflow.”

“大多数开发者需要：2-3 个 MCP server（GitHub、文件系统、一个领域相关的）+ 几个自定义 Skills，基本就够用了。”

Claude Code’s tool search already optimizes for this:

Claude Code 的工具搜索已经把这件事考虑进去了：

  • Lazy loading: When MCP tools exceed 10% of context, it automatically switches to lazy mode

  • 85-95% token savings: Metadata only loads initially, full definitions on demand

  • 延迟加载：当 MCP 工具占到上下文 10% 以上时，自动切到延迟模式

  • 节省 85-95% token：一开始只加载元数据，需要时再拉完整定义

This is exactly what the Chinese article’s “72% context” problem needed: not abandoning MCP, but smarter loading.

这其实正好对上了那篇中文文章说的“72% 上下文”问题：不是把 MCP 扔掉，而是换一种更聪明的加载方式。

What the Debate Misses | 争论忽略的是什么

The “MCP is dead” narrative focuses on the wrong layer. MCP is a toolbox protocol — it’s not supposed to tell you when to use a hammer vs screwdriver. That’s the job of:

“MCP 已死”这个说法盯错了层面。MCP 本来就是个工具箱协议，它不负责告诉你什么时候该拿锤子、什么时候该拿螺丝刀。这个活儿该由下面这些层来补：

  1. Prompt engineering — explaining tool selection logic | 提示工程——解释工具选择逻辑
  2. Skills — encoding team-specific procedures | Skills——把团队自己的做法写成流程
  3. Agent orchestration — intelligent tool routing | agent编排——让 agent 自己选路、自己分流

Microsoft Research found that Anthropic used a tool testing agent to rewrite defective MCP prompts, improving task completion by 40%. The issue isn’t MCP — it’s that many MCP server descriptions are poorly written.

微软研究院发现，Anthropic 用了一个工具测试 agent 去重写有问题的 MCP 提示词，把任务完成效率**提升了40%**。问题不在 MCP 本身，而是很多 MCP server 的描述写得太差。

The fix isn’t abandoning MCP. It’s adding the procedural layer on top.

解决方法不是放弃 MCP，而是得在上面把“流程”这层给补齐。

The Bottom Line | 结论

MCP isn’t being abandoned. It’s just being put in the right box: MCP handles the toolbox layer, Skills handle the recipe layer.

MCP 没有被抛弃，只是回到了它该在的位置：MCP 负责工具箱这一层，Skills 负责菜谱这一层。

The companies still investing in MCP — Pinterest, Microsoft, Google Cloud — show that the protocol is not dead. It’s one layer in a two-layer system.

Pinterest、Microsoft、Google Cloud 还在继续投 MCP，说明这东西没死。只是大家后来才看清，它只是这套“双层架构”里的基础设施层。

Toolbox without recipes is just a pile of tools.

没有菜谱的工具箱，本质上就是一堆工具。

The MCP conversation was never MCP vs CLI. It’s MCP + Skills — and most people haven’t discovered Skills yet.

MCP 这场讨论从来就不是 MCP vs CLI，而是MCP + Skills。问题只在于，大多数人还没把 Skills 这层补上。
