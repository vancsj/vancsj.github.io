MCP: Rise, Limits, and What's Next

发表于 2026-03-04 阅读次数：

In November 2024, Anthropic open-sourced the Model Context Protocol. One year later it had 97 million monthly SDK downloads, 10,000+ public servers, and commitments from OpenAI and Google. Then Anthropic gave it away to the Linux Foundation.

2024年11月，Anthropic开源了模型上下文协议（MCP）。一年后，它拿下了9700万月SDK下载量、10000多个公开服务器，连OpenAI和Google都公开宣布支持。然后Anthropic把它捐给了Linux基金会。

Three questions worth answering: what problem MCP actually solved, where it hits walls in production, and what Anthropic is betting on next.

三个值得回答的问题：MCP到底解决了什么问题，它在生产环境里碰到了什么墙，以及Anthropic接下来押注什么。

The N×M Problem

N×M问题

Before MCP, connecting AI to external tools was an N×M problem. Every new model (N) talking to every new data source (M) needed custom integration code. Three models times five tools equals fifteen bespoke adapters. That doesn’t scale.

MCP之前，AI接外部工具是个N×M问题。每个新模型（N）和每个新数据源（M）之间都要写一套定制集成代码。三个模型乘以五个工具就是十五套适配器。这没法扩展。

MCP’s fix is straightforward: a standard client-server protocol built on JSON-RPC 2.0. Tool providers implement an MCP server once; every MCP-compatible AI client can use it. The model never touches the external system directly — the client acts as intermediary, forwarding structured call instructions to the server and relaying results back.

MCP的办法很直接：定义一个基于JSON-RPC 2.0的标准客户端-服务器协议。工具提供方实现一次MCP服务器，所有兼容MCP的AI客户端都能用。模型不直接接触外部系统——客户端做中间人，把结构化调用指令转发给服务器，再把结果传回来。

Two concrete benefits. First, no vendor lock-in. Whether you run Claude, GPT, or an open-source model on the backend, the tool integration code stays the same. Second, sensitive data stays inside the enterprise firewall — only processed results go back to the model.

两个实打实的好处。第一，不锁定供应商——后端用Claude还是GPT还是开源模型，工具集成代码不用改。第二，敏感数据可以留在企业自己的防火墙内，只有处理后的结果返回给模型。

On the server side, MCP abstracts capabilities into three core primitives: Resources (data access), Tools (action execution), and Prompts (reusable workflow templates). There’s also a client-side primitive called Sampling that lets servers request completions from the model — but it’s a client capability, not something servers expose. Some writeups lump all four together at the same level; that’s architecturally inaccurate.

在服务器端，MCP把能力抽象成三个核心原语：资源（Resources，数据访问）、工具（Tools，执行动作）、提示词模板（Prompts，可复用的工作流）。另外还有一个客户端侧的原语叫采样（Sampling），允许服务器反过来请求模型做补全——但这是客户端能力，不是服务器暴露的。有些文章把这四个放在同一层级，在架构上是不准确的。

SDKs cover the major languages. TypeScript and Python are Tier 1 alongside C# and Go. Java is Tier 2. What used to take weeks of integration work now takes hours for a prototype.

SDK覆盖了主流语言。TypeScript和Python是第一梯队，C#和Go紧随其后，Java是第二梯队。以前需要几周的集成工作，现在几小时就能出原型。

How It Took Over in One Year

一年内如何统一行业

The origin story is mundane in the best way. David Soria Parra, a Member of Technical Staff at Anthropic, got tired of copying artifacts between Claude Desktop (good visualization, no external access) and his code editor (full code access, poor visualization). He and colleague Justin Spahr-Summers built a general-purpose context injection standard. The internal codename was “Claude Connect.”

起源故事平淡得恰到好处。Anthropic的工程师David Soria Parra受够了在Claude桌面端（可视化好，但没有外部访问权限）和代码编辑器（能访问代码，但可视化差）之间反复复制粘贴。他和同事Justin Spahr-Summers搞了一套通用的上下文注入标准，内部代号叫”Claude Connect”。

November 2024 — Anthropic open-sources MCP with Python and TypeScript SDKs.

2024年11月 — Anthropic开源MCP，附带Python和TypeScript SDK。

March 26, 2025 — Sam Altman posts on X: “People love MCP and we are excited to add support across our products.” OpenAI is in.

2025年3月26日 — Sam Altman在X上发帖：”People love MCP and we are excited to add support across our products。”OpenAI入局。

April 9, 2025 — Demis Hassabis follows: “MCP is a good protocol and it’s rapidly becoming an open standard for the AI agentic era.” Google confirms Gemini support. All three major model providers now on board.

2025年4月9日 — Demis Hassabis跟进：”MCP is a good protocol and it’s rapidly becoming an open standard for the AI agentic era。”Google确认Gemini支持MCP。三大模型厂商到齐。

June 2025 — Salesforce launches Agentforce 3 with a built-in MCP client and 30+ partner MCP servers. Enterprise market starts moving.

2025年6月 — Salesforce发布Agentforce 3，内置MCP客户端，30多个合作伙伴MCP服务器上线。企业级市场开始跟进。

November 2025 — The anniversary spec update (2025-11-25 version) adds async task processing, standardized OAuth 2.0 authorization flows, and enterprise authorization extensions. SEP-1577 enhances the Sampling mechanism with tool-use support, letting MCP servers run agentic loops via the client’s LLM. (Note: the reverse-calling capability itself predated this — SEP-1577 specifically added tool calling within sampling requests.)

2025年11月 — 一周年规范更新（2025-11-25版本），加入异步任务处理、标准化OAuth 2.0授权流程和企业级授权扩展。SEP-1577增强了采样机制的工具调用支持，让MCP服务器能通过客户端的大模型运行代理循环。（注意：反向调用能力本身在此之前就有——SEP-1577具体增加的是采样请求中的工具调用支持。）

December 9, 2025 — Anthropic donates MCP to the Linux Foundation’s new Agentic AI Foundation (AAIF), alongside OpenAI’s AGENTS.md and Block’s goose. AWS, Microsoft, Google, Cloudflare, and Bloomberg are all supporting members.

2025年12月9日 — Anthropic把MCP捐给Linux基金会新成立的Agentic AI Foundation（AAIF），和OpenAI的AGENTS.md、Block的goose并列为创始项目。AWS、微软、Google、Cloudflare、Bloomberg都是支持方。

From internal tool to industry standard in twelve months. The network effects were real: tool providers implement once and reach every major AI platform; AI developers plug in one client and get access to thousands of tools. Any competitor trying to maintain a walled garden faced rapid marginalization.

从内部工具到行业标准，十二个月。网络效应是实打实的：工具提供商只需实现一次就能触达所有主流AI平台；AI开发者接入一个客户端就能调用数千种工具。任何试图维持封闭花园的竞争者都面临快速边缘化。

Where It Hits Walls

碰到的墙

MCP demos well. The problems show up at production scale.

MCP做演示很好看。问题出在生产规模上。

The Context Window Tax

上下文窗口税

MCP’s standard flow requires loading all available tool definitions — names, descriptions, input parameter JSON Schemas, return formats — into the model’s system prompt at conversation initialization. With three to five tools, that’s a few hundred tokens. Negligible.

MCP的标准流程要求在对话初始化时，把所有可用工具的定义——名称、描述、输入参数的JSON Schema、返回格式——全部塞进模型的系统提示词。连三五个工具的时候，只有几百个token，无所谓。

Enterprise environments don’t have three to five tools. Cloudflare has over 2,500 API endpoints. Their engineering team measured it: exposing all endpoints as MCP tools in the standard way consumes 1.17 million tokens just for tool declarations. The model hasn’t started thinking about the user’s question yet, and the context window is already mostly full.

企业环境不是三五个工具。Cloudflare有2500多个API端点。他们的工程团队测过：按标准MCP模式把所有端点暴露给模型，光工具声明就要吃掉117万个token。模型还没开始”想”用户的问题，上下文窗口就已经大半满了。

Then there’s the intermediate result problem. If an agent reads a 100-page document from Google Drive and needs to write it to Salesforce, the entire document text flows through the model’s context window — model reads it in, generates it back out as input for the next tool call. The document doesn’t need the model to “understand” it; it’s just being shuttled. But tokens are consumed both ways.

还有中间结果的问题。如果代理要从Google Drive读一份100页的文档然后写到Salesforce里，整份文档的文本要经过模型的上下文窗口——模型读进来，再生成出去作为下一个工具调用的输入。文档不需要模型”理解”，它只是在做搬运，但token两头都在烧。

Security: Not Theoretical

安全：不是理论风险

Between 2025 and early 2026, the MCP ecosystem produced a series of real security incidents — not hypothetical scenarios.

2025年到2026年初，MCP生态暴露出了一系列真实的安全事故——不是假设场景。

Published CVEs:

  • CVE-2025-6514 (CVSS 9.6): The mcp-remote npm package allowed remote code execution via OS commands injected through OAuth discovery fields
  • CVE-2025-68143 / 68145: Anthropic’s own mcp-server-git had path validation bypass flaws — arbitrary repository creation and restricted file access. Reported September 2025, fixed December 2025
  • CVE-2025-59536: RCE and API token exfiltration through Claude Code project files

已公开的CVE：

  • CVE-2025-6514（CVSS 9.6）：mcp-remote npm包允许通过OAuth发现字段注入操作系统命令来远程执行代码
  • CVE-2025-68143/68145：Anthropic自己维护的mcp-server-git存在路径验证绕过——可在任意路径创建仓库和访问受限文件。2025年9月报告，12月才修复
  • CVE-2025-59536：通过Claude Code项目文件实现RCE和API token外泄

Real incidents:

  • Asana (June 2025): An MCP-powered feature caused customer data to bleed across MCP instances. The feature was pulled offline for two weeks.
  • Supabase (mid-2025): A Cursor agent holding privileged service-role access processed user-supplied input as commands. Attackers embedded SQL to exfiltrate integration tokens.

真实事故：

  • Asana（2025年6月）：MCP驱动的功能导致客户数据在不同MCP实例间泄漏，整个功能下线两周。
  • Supabase（2025年中）：Cursor代理持有高权限的service-role访问，攻击者通过用户输入嵌入SQL指令，窃取了集成token。

Systemic audit: An analysis of 2,614 MCP implementations found 82% using filesystem operations vulnerable to path traversal (CWE-22), 67% with code injection risk (CWE-94), and 34% with command injection risk (CWE-78).

系统性审计： 对2614个MCP实现的分析显示，82%使用了容易受路径遍历攻击的文件系统操作（CWE-22），67%存在代码注入风险（CWE-94），34%存在命令注入风险（CWE-78）。

Prompt injection is the deeper structural issue. MCP server tool descriptions go directly into the model’s system instructions. If an attacker can tamper with data the agent reads — a hidden instruction in a webpage, PDF, or email — they can hijack the model into abandoning its original task and executing unauthorized operations. Traditional security tools can’t catch this because the malicious instruction is indistinguishable in format from legitimate data.

提示词注入是更深层的结构性问题。MCP服务器的工具描述直接进入模型的系统指令。如果攻击者能篡改代理要读取的数据——在网页、PDF或邮件里藏一段指令——就能劫持模型放弃原任务，执行未授权的操作。传统安全工具拦不住这个，因为恶意指令和正常数据在格式上没有区别。

The November 2025 spec introduced OAuth improvements and enterprise authorization extensions, which helped with transport-layer risks. But the architectural problem — a single MCP server holding OAuth tokens for Salesforce, Google Workspace, and Slack simultaneously, becoming a high-value target — hasn’t been fundamentally resolved.

2025年11月的规范更新引入了OAuth改进和企业授权扩展，缓解了传输层面的风险。但架构上的问题——一个MCP服务器同时持有Salesforce、Google Workspace、Slack的OAuth令牌，成为高价值攻击目标——没有根本解决。

And then there was the supply chain attack on the ClawHub skills marketplace in January 2026. Initial scans found 341 malicious skills across 2,857 packages; expanded scans revealed over 824. Bitdefender confirmed roughly 20% of deeply analyzed packages contained malicious payloads. The tool ecosystem’s growth outpaced its security hardening — significantly.

2026年1月还发生了ClawHub技能市场的供应链攻击。初次扫描在2857个包中发现了341个恶意技能；扩大扫描后发现超过824个。Bitdefender确认深度分析的包中约20%包含恶意载荷。工具生态的增长速度远远跑在了安全加固前面。

Too Many Tools, Wrong Choices

工具太多，选错了

MCP routing depends on natural language tool descriptions to guide model selection. Few tools — no problem. Many tools with even slight semantic overlap in their descriptions — the model picks the wrong one.

MCP的路由依赖自然语言工具描述来引导模型选择。工具少的时候没问题。工具一多，描述之间只要有一丝语义重叠，模型就容易调错。

Development teams end up iterating on tool descriptions, hardcoding call examples — essentially doing prompt engineering to compensate for the protocol’s limitations. The “plug and play” promise turns into ongoing maintenance overhead.

开发团队最后不得不反复修改工具描述、硬编码调用示例——本质上是在做提示词工程来弥补协议的不足。”即插即用”的承诺变成了持续的维护负担。

What Anthropic Is Betting On

Anthropic在押注什么

A common misconception worth clearing up: Anthropic did not “abandon” or “deprioritize” MCP. Agent Skills and Code Execution are higher-level abstractions built on top of MCP, not replacements. MCP remains the tool-connectivity plumbing. Anthropic’s position is that plumbing alone isn’t enough — you also need the layers above it.

一个值得澄清的误解：Anthropic并没有”放弃”或”降级”MCP。 Agent Skills和Code Execution是构建在MCP之上的更高层抽象，不是替代品。MCP仍然是底层的工具连接管道。Anthropic的观点是，光有管道不够——还需要管道之上的东西。

Agent Skills: On-Demand Knowledge Containers

Agent Skills：按需加载的知识容器

Introduced October 2025, open-standardized December 2025. The core problem Skills solve is context consumption.

2025年10月推出，12月作为开放标准发布。Skills要解决的核心问题是上下文消耗。

Each skill is defined by a SKILL.md file with a three-level loading mechanism:

每个技能由一个SKILL.md文件定义，采用三级加载机制：

Level 1: At startup, only the skill’s name and a one-line description (from YAML frontmatter) get loaded — roughly a few dozen tokens per skill. The agent “knows” what skills it has without consuming meaningful context.

第一级： 启动时只加载技能的名称和一句话描述（来自YAML frontmatter），每个技能大约几十个token。代理”知道”自己有哪些技能，但几乎不占上下文。

Level 2: When the agent determines the current task needs a specific skill, it reads the SKILL.md body from the filesystem on demand — operational specs, workflow steps, typically a few thousand tokens. Irrelevant skills never enter the context window.

第二级： 代理判断当前任务需要某个技能时，通过文件系统按需读取SKILL.md的主体内容——操作规范、工作流步骤，通常几千个token。不相关的技能永远不进上下文。

Level 3: Large datasets, API reference docs, and Python/Bash scripts bundled with the skill are accessed only when explicitly needed. Scripts execute in a sandbox; their source code and intermediate processing never enter the model’s context. Only the final result comes back.

第三级： 技能包里的大型数据集、API参考文档和Python/Bash脚本，只在明确需要时才被访问。脚本在沙盒里执行，源代码和中间过程不进模型上下文，只有最终结果返回。

To be honest about what this is: it’s on-demand file reading plus local script execution. Smart engineering, not a paradigm shift. Its value is clear — MCP front-loads all tool definitions into the context; Skills break information into layers loaded incrementally — but calling it a “cognitive architecture revolution” overstates it.

说白了这是什么：按需读文件加上本地脚本执行。工程上很聪明，但不是范式转移。它的价值很明确——MCP把所有工具定义一次性塞进上下文，Skills把信息拆成层级按需加载——但把它叫”认知架构革命”就夸大了。

The division of labor: MCP handles dynamic data fetching that requires network connections (pulling Slack messages, querying databases). Skills tell the agent how to process that data (what report template to use, what approval workflow to follow).

两者的分工：MCP负责需要网络连接的动态数据获取（拉Slack消息、查数据库），Skills告诉代理怎么处理这些数据（用什么报告模板、走什么审批流程）。

Pre-built skills already exist for PowerPoint, Excel, Word, and PDF generation. Custom skills can be authored by anyone — it’s just a markdown file with frontmatter and optional scripts.

预置技能已经有PowerPoint、Excel、Word和PDF生成。自定义技能任何人都能写——就是一个带frontmatter的markdown文件加上可选的脚本。

Code Execution: Write Code, Don’t Call Tools

代码执行：写代码，别调工具

Both Anthropic and Cloudflare are pushing the same direction: let agents write code to orchestrate tool calls instead of calling MCP tools one by one.

Anthropic和Cloudflare都在推同一个方向：让代理写代码来编排工具调用，而不是一个个调MCP工具。

The logic is simple. Modern LLMs absorbed billions of lines of real code during pre-training. They write TypeScript and Python fluently. But JSON-formatted tool-calling instructions? Far less training data. Forcing the model to work in an unfamiliar format is inherently less efficient. Cloudflare’s engineering team put it vividly: it’s like sending Shakespeare to a one-month crash course in a foreign language and expecting a masterpiece.

逻辑很简单。现代大模型在预训练阶段吸收了数十亿行真实代码，写TypeScript和Python极其流畅。但JSON格式的工具调用指令？训练数据少得多。强迫模型用不熟练的格式做事，效率当然低。Cloudflare的工程团队打了个比方：这就像把莎士比亚送去学一个月外语速成班，然后指望他用外语写传世名篇。

In code execution mode, MCP server endpoints become importable modules — import salesforce; salesforce.query(...). Instead of calling tools one at a time and waiting for results to flow back through the context, the agent writes a script: pull 100,000 records into a local variable via API, filter down to the 5 matching records using code logic, print only those 5 back to the model.

在代码执行模式下，MCP服务器的端点变成可导入的代码模块——import salesforce; salesforce.query(...)。代理不再一个个调工具等结果回来，而是写一段脚本：通过API把10万条记录拉到本地变量里，用代码逻辑筛选出符合条件的5条，只把这5条打印回模型。

Anthropic’s measured result: a workflow went from 150,000 tokens to 2,000 tokens — 98.7% reduction. Cloudflare’s numbers were even more dramatic: 1.17 million tokens down to roughly 1,000. These are best-case figures from specific test scenarios, not averages. But the order-of-magnitude improvement is real.

Anthropic的测试数据：一个工作流从150,000 token降到2,000 token——节省98.7%。Cloudflare的数据更夸张：从117万token降到约1,000。这些都是特定测试场景的最优数字，不是平均值。 但数量级的改善是真实的。

Sandbox security relies on OS-level isolation — bubblewrap on Linux, seatbelt on macOS — enforcing dual physical isolation on both filesystem and network traffic. An additional benefit: sensitive data like PII stays in the execution sandbox and never reaches the model.

沙盒安全靠操作系统级别的隔离——Linux上用bubblewrap，macOS上用seatbelt——对文件系统和网络做双重物理隔离。额外的好处：PII等敏感数据留在执行沙盒里，不会到达模型。

Why Give MCP Away

为什么捐出MCP

Donating MCP to the Linux Foundation wasn’t abandonment. It was acknowledging reality: connection protocols commoditize. HTTP became the universal communication standard. SQL became the universal query language. MCP was already well on its way to becoming the universal tool-connection protocol. Trying to hold onto it as a proprietary moat would have been futile and counterproductive.

把MCP捐给Linux基金会不是放弃，是认清现实：连接协议会商品化。 HTTP成了通用通信标准，SQL成了通用查询语言，MCP已经在成为通用工具连接协议的路上了。试图把它攥在手里当护城河，既徒劳又适得其反。

The donation achieved two things. First, it pulled OpenAI into the same open-standard framework (OpenAI contributed AGENTS.md), preventing competitors from locking developers into proprietary protocol stacks. As long as tool integration standards are open, customers can swap models freely — and Anthropic bets its models are stronger. Second, it offloaded the maintenance burden of a sprawling protocol ecosystem to the community and cloud giants, freeing Anthropic’s scarce research talent to focus on model cognition.

捐赠做成了两件事。第一，把OpenAI拉进同一个开放标准框架（OpenAI捐了AGENTS.md），防止对手用私有协议锁定开发者。只要工具接入标准是开放的，客户就能随时换模型——而Anthropic赌的是自己模型更强。第二，把庞大协议生态的维护重担甩给社区和云厂商，自己的研发资源集中在模型认知能力上。

The Bigger Picture in 2026

2026年的更大图景

MCP is no longer the only protocol on the field.

MCP不再是场上唯一的协议了。

WebMCP: The Browser Frontier

WebMCP：浏览器前线

If MCP solved backend tool integration, WebMCP targets the frontend. It’s a JavaScript API (navigator.modelContext) that lets web developers expose their application functionality as tools — JavaScript functions with natural language descriptions and structured schemas that AI agents can invoke directly.

如果说MCP解决了后端工具集成，WebMCP瞄准的是前端。它是一个JavaScript API（navigator.modelContext），让网站开发者把应用功能暴露为工具——带有自然语言描述和结构化schema的JavaScript函数，AI代理可以直接调用。

It’s being developed by the W3C Web Machine Learning Community Group, with editors from Microsoft and Google. Chrome 146 Canary shipped early preview support in February 2026. Important caveat: it’s a W3C Draft Community Group Report, not a W3C Standard and not on the standards track. It doesn’t carry formal W3C endorsement. Calling it a “W3C standard” in articles, as some have, is premature.

它由W3C Web Machine Learning社区组开发，编辑来自微软和Google。Chrome 146 Canary在2026年2月发布了早期预览支持。重要的限定：这是W3C社区组草案报告，不是W3C标准，也不在标准化轨道上。 它没有W3C的正式背书。有些文章把它叫”W3C标准”，为时过早。

The promise is replacing brittle screen-scraping with reliable, schema-driven tool interaction. If it takes hold, it would make existing websites “agent-ready” without backend changes.

它的前景是用可靠的、schema驱动的工具交互取代脆弱的屏幕抓取。如果能站稳脚跟，现有网站无需后端改造就能”代理就绪”。

A2A: Agent-to-Agent Communication

A2A：代理间通信

Google announced the Agent-to-Agent protocol in April 2025. Where MCP connects agents to tools (vertical), A2A connects agents to other agents (horizontal) — task negotiation, state synchronization, conflict resolution between heterogeneous agent systems.

Google在2025年4月推出了Agent-to-Agent协议。MCP连接代理和工具（纵向），A2A连接代理和其他代理（横向）——异构代理系统之间的任务协商、状态同步和冲突解决。

It launched with 50+ partners, hit 150+ supported organizations by v0.3 (July 2025), and was also donated to the Linux Foundation’s AAIF in December 2025. Some developers have questioned whether A2A genuinely differentiates from MCP — “I truly don’t understand what A2A is bringing that is not achievable with MCP” was a representative comment. That boundary debate is still ongoing.

它发布时有50多个合作伙伴，到v0.3（2025年7月）达到150多个支持组织，同样在2025年12月捐给了Linux基金会的AAIF。有开发者质疑A2A是否真的和MCP有本质区别——“I truly don’t understand what A2A is bringing that is not achievable with MCP”是一个有代表性的评论。这个边界争论还在继续。

The Competitive Landscape Is Wider Than Two Players

竞争格局远不止两家

Most coverage frames the competition as Anthropic vs. OpenAI. That misses half the field.

大部分报道把竞争格局简化为Anthropic对OpenAI。这漏掉了半个战场。

Microsoft merged AutoGen and Semantic Kernel into a unified Agent Framework in October 2025 — supporting C#, Python, and Java with Azure integration and SOC 2/HIPAA compliance. GA targeting Q1 2026. This is a serious enterprise play.

微软 在2025年10月把AutoGen和Semantic Kernel合并为统一的Agent Framework——支持C#、Python和Java，集成Azure，符合SOC 2/HIPAA。目标2026年Q1 GA。这是认真的企业级布局。

LangChain/LangGraph remains the largest community by adoption, with graph-based workflow orchestration. CrewAI has carved out a niche in role-based multi-agent collaboration. Google’s ADK integrates A2A natively with Vertex AI.

LangChain/LangGraph 按采纳量仍是最大的社区，主打基于图的工作流编排。CrewAI 在基于角色的多代理协作领域站稳了脚跟。Google的ADK 原生集成A2A和Vertex AI。

The agent framework space has five to six serious contenders, not two.

代理框架领域有五六个认真的竞争者，不是两个。

Claude Opus 4.6: What the Numbers Actually Show

Claude Opus 4.6：数字到底说明了什么

Released February 5, 2026. The headline specs: 1 million token context window (beta), 128K output tokens, adaptive thinking with effort controls, and a native Agent Teams feature for parallel Claude instances sharing a codebase.

2026年2月5日发布。主要参数：100万token上下文窗口（beta），128K输出token，自适应思考（可控力度），原生Agent Teams功能支持多个Claude实例在共享代码库上并行工作。

The stress test that got the most attention: 16 parallel Claude agents, each in its own Docker container sharing a Git repo, produced 100,000 lines of Rust implementing a C compiler. It can boot Linux 6.9 on x86/ARM/RISC-V, compile QEMU, FFmpeg, SQLite, PostgreSQL, and Redis, pass 99% of GCC’s torture test suite, and run Doom.

最受关注的压力测试：16个并行Claude代理，各自在独立Docker容器中共享一个Git仓库，产出了10万行Rust代码，实现了一个C编译器。它能在x86/ARM/RISC-V上启动Linux 6.9，编译QEMU、FFmpeg、SQLite、PostgreSQL和Redis，通过GCC torture测试套件的99%，还能跑Doom。

The numbers behind it: ~$20,000 in API spend, 2 billion input tokens, 140 million output tokens, roughly 2,000 Claude Code sessions over two weeks.

背后的数字：约2万美元API费用，20亿输入token，1.4亿输出token，大约2000次Claude Code会话，历时两周。

The caveats that don’t make the headlines: it requires GCC for 16-bit x86 real-mode code, lacks its own assembler and linker (uses GCC’s), generates less efficient code than GCC, and cannot replace production compilers. The coordination mechanism was file-based locking — text files as task locks — not some sophisticated multi-agent protocol.

不上头条的限定条件：它需要GCC来处理16位x86实模式代码，没有自己的汇编器和链接器（用GCC的），生成代码效率低于GCC，无法替代生产编译器。协调机制是基于文件的锁——用文本文件做任务锁——不是什么精妙的多代理协议。

It’s an impressive demonstration of what parallel LLM agents can produce in a controlled environment. Extrapolating it to “AI teams will replace software engineers” would be premature.

这是一个令人印象深刻的演示，展示了并行LLM代理在受控环境下能做什么。从这里推断”AI团队将取代软件工程师”还为时过早。

How to Think About This

怎么看这件事

MCP accomplished something genuinely difficult: getting an entire industry — including direct competitors — to adopt a single tool-connection standard in one year. That’s real.

MCP做成了一件很难的事：让整个行业——包括直接竞争对手——在一年内采纳同一套工具连接标准。这是实打实的成就。

But MCP’s success is precisely what made it invisible. When everyone supports the same protocol, the protocol itself stops being a competitive advantage and becomes infrastructure — essential but unremarkable, like TCP/IP. Anthropic read this correctly and shifted focus to higher layers: Skills (teaching agents how to do things) and Code Execution (making agents do things more efficiently).

但MCP的成功恰恰让它变得”隐形”。当所有人都支持同一个协议，协议本身就不再是竞争优势，而是基础设施——不可或缺但不引人注目，像TCP/IP一样。Anthropic看清了这一点，把精力转向更高的层次：Skills（教代理怎么做事）和Code Execution（让代理更高效地做事）。

Anthropic’s own narrative about these shifts deserves discount, though. Skills’ “progressive disclosure” is fundamentally reading files on demand. Code Execution’s 98.7% savings is a cherry-picked best case. These are solid engineering improvements — calling them “paradigm shifts” or “dimensional strikes” is marketing.

不过，Anthropic自己对这些转变的叙述需要打折理解。Skills的”渐进式披露”本质上是按需读文件。Code Execution的98.7%节省率是精心挑选的最优场景。这些是扎实的工程改进——把它们叫”范式转移”或”降维打击”是营销话术。

The issue that should get more attention is security. MCP adoption massively outpaced security hardening. Of 2,614 analyzed implementations, 82% have path traversal vulnerabilities. Real data leaks have already happened (Asana, Supabase). Supply chain attacks have already hit the ecosystem (ClawHub). Before agent systems get more autonomy, this gap needs closing — and that urgency gets lost in the excitement about what agents can do.

真正值得更多关注的是安全。MCP的采纳速度远远跑在了安全加固前面。2614个被分析的实现中，82%存在路径遍历漏洞。真实的数据泄漏已经发生（Asana、Supabase）。供应链攻击已经命中生态（ClawHub）。在代理系统获得更多自主权之前，这个缺口必须补上——而这种紧迫感在对代理能力的兴奋中被淹没了。

The competitive landscape has matured into a layered stack with clear boundaries. MCP handles agent-to-tool connectivity. A2A handles agent-to-agent coordination. WebMCP handles browser-side exposure. Skills handle domain knowledge. Agent frameworks (LangChain, CrewAI, Microsoft Agent Framework, OpenAI Agents SDK, Google ADK) handle orchestration. No single protocol or framework covers the whole picture, and anyone claiming otherwise is selling something.

竞争格局已经成熟为一个边界清晰的分层栈。MCP处理代理到工具的连接。A2A处理代理到代理的协调。WebMCP处理浏览器端暴露。Skills处理领域知识。代理框架（LangChain、CrewAI、微软Agent Framework、OpenAI Agents SDK、Google ADK）处理编排。没有哪个单一协议或框架能覆盖全貌——谁这么说，谁就是在卖东西。

Where the real competition is heading: not “how to call tools” but how to build models that can coordinate multi-agent teams across million-token contexts, plan over long horizons, and self-correct when things go wrong — all within hard security boundaries. That’s where the defensible moats are. MCP was the necessary foundation. What gets built on it is where the game is.

真正的竞争方向：不是”怎么调工具”，而是怎么构建能够在百万级token上下文中协调多代理团队、进行长期规划、在出错时自我纠正的模型——同时在严格的安全边界内。可防御的护城河在那里。MCP是必要的地基。在地基之上建什么，才是真正的较量。
